# 로드맵용 운영 원칙 (☐ 로그인·외부 API 도입 시 활성화)

> **상태: 냉동(frozen).** 현재 swithme는 백엔드·DB·외부 API가 없어 아래 규칙의 적용 대상이 **하나도 없다**. 로그인·계정 동기화(백엔드/DB/인증 계층)가 도입되는 시점에 활성화한다.
>
> 이 문서는 상시 로드되는 `CLAUDE.md`에서 분리되었다. 활성 규칙만 `CLAUDE.md`에 두고, 미래용 규칙은 여기 보존하여 매 세션 로드 비용을 줄인다. 로드맵 진입 시 이 내용을 `CLAUDE.md`로 되살린다.

---

## 3계층 아키텍처 — 실행 계층(Execution)의 로드맵 확장

현재 `execution/`은 메타 추출 스크립트 전용이다. 로드맵 도입 시 다음이 추가된다:

- 환경 변수 및 API 토큰은 `.env` 파일에서 참조한다.
- 백엔드 API 호출, 데이터 처리, 파일 시스템 제어, 데이터베이스 상호작용 등을 결정론적 스크립트로 위임할 수 있다.
- 수동 작업을 대체하는 빠르고 테스트 가능한 코드이며, 명확한 주석을 포함한다.

---

## 운영 원칙

### 1. 기존 도구 우선 확인
새로운 스크립트를 작성하기 전, 반드시 해당 지침에 명시된 도구가 `execution/` 디렉토리에 존재하는지 먼저 확인할 것.

### 2. 오류 발생 시 자가 수정 (Self-Correction)
- 오류 메시지와 스택 트레이스를 분석하여 원인을 파악한다.
- 스크립트를 수정하고 재테스트한다.
  - **단, 유료 API 토큰이나 크레딧을 소모하는 스크립트의 경우 수정 전 사용자에게 승인을 요청할 것.**
- 새롭게 파악한 API 제약 사항(Rate limit 등), 타이밍 문제, 예외 상황을 해당 `directives/` 마크다운 파일에 업데이트한다.

### 3. 학습 데이터를 통한 지침 업데이트
지침은 고정된 문서가 아니다. 더 나은 로직, 자주 발생하는 오류에 대한 방어 코드, API 구조 변경 등을 발견하면 즉시 해당 지침 파일을 업데이트할 것. **단, 사용자의 명시적 지시 없이 기존 지침의 핵심 목표를 임의로 삭제하거나 덮어쓰지 않는다.**

### 자가 수정 루프 프로토콜
1. **수정** — 스크립트 코드의 오류를 수정한다.
2. **검증** — 도구가 정상 동작하는지 터미널에서 테스트한다.
3. **문서화** — 발견된 문제와 해결된 새로운 작업 흐름을 지침 문서에 반영한다.
