# CLAUDE.md

> **글로벌 규칙 함께 적용:** Claude Code는 `~/.claude/CLAUDE.md`(페르소나·언어 규칙·작업 공정·Plan 운영·문서 탐색·시크릿 보안·신규 규칙 위치 결정 원칙 등 보편 규칙)도 함께 로드한다. 본 파일은 **swithme 고유 규칙만** 담는다. 충돌 시 **이 파일이 글로벌을 덮어쓴다**.
>
> **신규 규칙 추가 시:** 글로벌 `~/.claude/CLAUDE.md`의 "신규 규칙·인사이트 추가 시 — 위치 결정 원칙"을 따라 글로벌 vs 프로젝트 위치를 먼저 판단한 뒤 기록한다.

## 프로젝트 정체성

swithme는 학습·활동 시간을 추적·분석하는 PWA이다.

- **현재(2026-04 기준):** Vanilla JS + HTML + CSS + `localStorage` 기반 **클라이언트 사이드 단독 동작**. 백엔드·DB·외부 API 호출 없음.
- **로드맵(미정 ☐):** 로그인·계정 동기화 도입 예정. 도입 시 백엔드/DB/인증 계층이 추가될 수 있음. 스택·인증 방식·DB는 미정.

> **현재/예정 구분:** 백엔드·API·DB·인증 관련 규칙(자가 수정 루프, 유료 API 토큰 승인, Rate limit 대응, `credentials.json/token.json` 등)은 **로드맵을 위해 보존된 규칙**이다. 현재 프로젝트엔 적용 대상이 없으니 향후 로그인 기능 도입 시 활성화한다.

## 항상 적용되는 핵심 규칙

매 작업마다 적용되는 swithme 고유 결정 규칙. 위반하면 즉시 깨진다.

- PC(`#desktop-shell`)와 모바일(`#mobile-shell`)은 별도 셸로 분리되어 있으므로, **UI 변경 시 양쪽 모두 반영**할 것.
- 상태는 `js/store.js`의 `state` 객체를 통해 중앙 관리되며, **변경 후 `saveToLocal()` 호출 필수**.
- 모바일은 PC 타이머를 `MutationObserver`로 감시하여 동기화하는 구조.
- 새 전역 함수는 `window`에 등록하는 기존 패턴을 따를 것.
- JS 코드 수정 후 `npm run lint`로 린트 검사를 수행할 것. 새로운 warning/error 추가 금지.
- 핵심 로직(`js/store.js`, `js/timer.js`) 변경 시 `npm run test:run`으로 기존 테스트 통과 여부를 확인할 것.

> 시크릿 보안(`.env` 사용·하드코딩 금지)은 글로벌 `~/.claude/CLAUDE.md` "보안" 섹션 참조.

## 에이전트 운영 — 3계층 아키텍처 (요약)

> 상세 내용은 본 섹션 그대로. 로드맵(백엔드/외부 API 도입) 시 활성화되는 운영 모델이다.

### 1계층: 지침 (Directive) — `directives/`

마크다운 형식의 표준 작업 절차(SOP). 작업의 목표·필수 입력값·도구·예상 출력·예외를 정의한다. **`directives/INDEX.md`를 먼저 읽고**, 작업에 필요한 지침 파일만 선택적으로 참조한다. 새 지침 파일 추가/삭제 시 `INDEX.md`도 함께 갱신.

### 2계층: 오케스트레이션 — Claude Code(당신)

`directives/`의 문서를 읽고, 올바른 순서로 도구를 호출하며, 오류를 처리하고, 학습한 내용을 바탕으로 지침을 업데이트한다. **직접 복잡한 데이터 처리/크롤링을 수행하지 않는다.** 지침을 읽고 `execution/`의 스크립트를 실행하여 입출력을 통제한다.

### 3계층: 실행 (Execution) — `execution/`

#### 현재 (✅ 사용 중)
- **메타 추출 스크립트 전용:** `extract-html-structure.py`, `extract-css-variables.py`, `extract-js-api.py`.
- 코드/HTML/CSS의 사실 정보(ID·셀렉터·함수 시그니처·아이콘 키 등)를 자동 추출하여 문서 동기화 비용을 제거한다.
- 비즈니스 로직 실행 계층이 **아니다**.

#### 로드맵 (☐ 로그인·동기화 도입 시)
- 환경 변수 및 API 토큰은 `.env` 파일에서 참조한다.
- 백엔드 API 호출, 데이터 처리, 파일 시스템 제어, 데이터베이스 상호작용 등을 결정론적 스크립트로 위임할 수 있다.
- 수동 작업을 대체하는 빠르고 테스트 가능한 코드이며, 명확한 주석을 포함한다.

> **아키텍처 목적:** 다단계 작업에서 확률적 모델의 오류 누적을 방지한다. 복잡한 로직은 결정론적 스크립트에 위임하고, Claude Code는 의사 결정과 흐름 제어에 집중한다.

## 로드맵용 운영 원칙 (☐ 로그인·외부 API 도입 시 활성화)

### 1. 기존 도구 우선 확인
새로운 스크립트를 작성하기 전, 반드시 해당 지침에 명시된 도구가 `execution/` 디렉토리에 존재하는지 먼저 확인할 것.

### 2. 오류 발생 시 자가 수정 (Self-Correction)
- 오류 메시지와 스택 트레이스를 분석하여 원인을 파악한다.
- 스크립트를 수정하고 재테스트한다.
  - **단, 유료 API 토큰이나 크레딧을 소모하는 스크립트의 경우 수정 전 사용자에게 승인을 요청할 것.**
- 새롭게 파악한 API 제약 사항(Rate limit 등), 타이밍 문제, 예외 상황을 해당 `directives/` 마크다운 파일에 업데이트한다.

### 3. 학습 데이터를 통한 지침 업데이트
지침은 고정된 문서가 아니다. 더 나은 로직, 자주 발생하는 오류에 대한 방어 코드, API 구조 변경 등을 발견하면 즉시 해당 지침 파일을 업데이트할 것. **단, 사용자의 명시적 지시 없이 기존 지침의 핵심 목표를 임의로 삭제하거나 덮어쓰지 않는다.**

### 자가 수정 루프 프로토콜
1. **수정** — 스크립트 코드의 오류를 수정한다.
2. **검증** — 도구가 정상 동작하는지 터미널에서 테스트한다.
3. **문서화** — 발견된 문제와 해결된 새로운 작업 흐름을 지침 문서에 반영한다.
