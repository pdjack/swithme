# UI/UX 일관성 가이드

> **문서 성격**: 규칙 문서 — PC와 모바일에서 같은 기능을 구현할 때 지켜야 할 원칙과 규칙을 정의한다.
> **참고**: 현재 코드의 구조·셀렉터·ID 매핑 등은 `STRUCTURE.md`, `style-reference.md`를 참조한다. 이 문서는 "어떻게 되어 있는가"가 아니라 **"앞으로 어떻게 만들어야 하는가"** 를 정의한다.

---

## 1. 디자인 원칙

### 1-1. 브랜드 아이덴티티

> 브랜드 컬러, CSS 변수, 카드 시스템 등의 상세 스펙은 [`style-reference.md`](style-reference.md)를 참조한다.

- **다크 모드 기반**: 모든 화면의 기본 배경은 다크(`--bg`)
- **글래스모피즘**: 카드·패널에 반투명 유리 질감(`--glass`, `--border`) 적용
- **타이포그래피**: 본문 `Inter`, 타이머·제목 `Outfit`
- **새 색상 추가 시**: 반드시 `:root` CSS 변수로 정의하고 `style-reference.md`에 기록

### 1-2. PC ↔ 모바일 일관성 기준

**기본 원칙: 같은 기능은 같은 경험을 제공한다.**

| 항목 | 일치 여부 | 설명 |
|------|-----------|------|
| **기능** | 반드시 일치 | 같은 기능은 양쪽에서 동일하게 동작해야 한다 |
| **시각 스타일** | 반드시 일치 | 같은 컴포넌트의 색상·폰트·여백·둥근 모서리는 동일 |
| **레이아웃** | 달라도 됨 | PC는 3열 그리드, 모바일은 1~2열 스택 등 화면 크기에 맞게 적응 |
| **인터랙션** | 플랫폼 적응 허용 | PC: 더블클릭/호버, 모바일: 롱프레스/스와이프 (단, 결과는 동일) |

> **판단 기준**: "PC에서 이 작업을 한 사용자가 모바일로 전환했을 때 혼란 없이 같은 작업을 할 수 있는가?"

## 2. 플랫폼 규칙

### 2-1. 텍스트 및 타이틀

- 모든 UI 텍스트(섹션 타이틀, 버튼 레이블, 빈 상태 메시지 등)는 **한국어**로 통일
- PC와 모바일에서 같은 섹션의 타이틀은 **동일한 문구** 사용

| 컴포넌트 | 통일 타이틀 |
|---------|-----------|
| 타임테이블 | 타임테이블 |
| 할 일 목록 | 할 일 |
| 타이머 | 타이머 / 스톱워치 |
| 캘린더 | 캘린더 |
| 분석 | 분석 |
| 설정 | 설정 |
| 회고 | 회고 |

### 2-2. 네이밍 컨벤션

#### HTML ID

| 플랫폼 | 규칙 | 예시 |
|--------|------|------|
| PC | 기능 기반 이름 | `#task-list`, `#timer-display` |
| 모바일 | `m-` 접두어 + 동일 이름 | `#m-task-list`, `#m-timer-display` |

#### CSS 클래스

| 플랫폼 | 규칙 | 예시 |
|--------|------|------|
| PC | 기능 기반 이름 | `.task-item`, `.tt-tab` |
| 모바일 | `m-` 접두어 + 동일 이름 | `.m-task-item`, `.m-tt-tab` |

- 상태 클래스는 플랫폼 접두어 없이 **동일한 이름** 사용: `active`, `completed`, `--active` 등
- BEM 변형(Modifier)은 `--` 사용: `.tt-tab--active`, `.m-tt-tab--active`

### 2-3. 이벤트 핸들링

**목표 패턴: 이벤트 위임 (`data-*` 속성 + `addEventListener`)**

```html
<!-- 목표 패턴 -->
<button data-delete-id="3">삭제</button>
```
```js
container.addEventListener('click', (e) => {
    const btn = e.target.closest('[data-delete-id]');
    if (btn) deleteTask(btn.dataset.deleteId);
});
```

**전환 정책: 점진적 전환**

| 상황 | 규칙 |
|------|------|
| 새 기능 구현 | 반드시 이벤트 위임 패턴 사용 |
| 기존 코드 수정 | 해당 컴포넌트를 수정할 때 함께 전환 |
| 기존 코드 유지 | 건드리지 않는 코드는 그대로 유지 |

### 2-4. 상태 동기화

**원칙: `store.js`의 `state` 객체가 단일 진실 공급원(Single Source of Truth)**

| 규칙 | 설명 |
|------|------|
| state 직접 참조 | PC/모바일 모두 `state` 객체를 직접 읽고 쓴다 |
| 변경 후 저장 | state 변경 후 반드시 `saveToLocal()` 호출 |
| 렌더 함수 호출 | state 변경 후 PC/모바일 렌더 함수를 모두 호출하여 양쪽 UI 갱신 |

**동기화 방식 우선순위**

1. **렌더 함수 직접 호출** (권장) — state 변경 후 `renderTasks()` + `renderMobileTasks()` 등 양쪽 렌더 함수 호출
2. **window 전역 함수 호출** (허용) — 모듈 간 순환 참조를 피해야 할 때
3. **MutationObserver** (최후 수단) — 직접 호출이 구조적으로 불가능한 경우에만. 반드시 스로틀링 적용

### 2-5. 모달 구현 규칙

| 규칙 | 설명 |
|------|------|
| 취소 버튼 필수 | 모든 모달에는 `취소` 또는 `닫기` 버튼을 반드시 포함한다 |
| 배치 일관성 | 버튼 순서: 좌측에 위험 동작(삭제), 중앙에 취소, 우측에 주요 동작(확인/수정) |
| 닫기 이벤트 정리 | 모달 닫힐 때 등록했던 이벤트 리스너를 모두 `removeEventListener`로 해제 |

### 2-6. 공통 컴포넌트 구현 규칙

**"데이터 로직은 공유, 렌더링은 분리"**

```
좋은 예:
  getGroupedTaskData()     ← 공유 (tasks.js에서 export)
  renderTasks()            ← PC 전용 렌더링
  renderMobileTasks()      ← 모바일 전용 렌더링

나쁜 예:
  renderTasks() 안에서 데이터 가공 + PC HTML 생성을 같이 수행
```

| 규칙 | 설명 |
|------|------|
| 데이터 가공 함수는 한 곳에서 정의 | `getGroupedTaskData()`, `buildSlotMap()` 등 |
| 렌더 함수는 플랫폼별로 분리 | PC/모바일의 HTML 구조가 다르므로 각각 작성 |
| 비즈니스 로직은 공유 | 삭제, 토글, 정렬 등의 로직은 한 곳에서 정의하고 양쪽에서 호출 |
