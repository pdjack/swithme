# 스윗미 (swithme) — 프로젝트 기획서 (PRD)

> **문서 성격**: 현행 상태 문서화 (v2.0)
> **최종 수정**: 2026-04-12

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **프로젝트명** | 스윗미 (swithme) |
| **한 줄 설명** | 유저가 체계적으로 할 일을 계획하고, 회고 데이터를 바탕으로 패턴을 분석하여 개선 방안을 제시하는 앱 |

### 목표 (Goal)

- 유저가 여러 분야(공부, 운동, 업무 등)에서의 계획이 분산되어 혼란해하지 않도록 한다.
- 단순히 계획을 하는 것에 그치지 않고, 회고 데이터 기반 분석과 처방전을 통해 **목표 이행률 상승**과 같은 성과와 발전을 이룰 수 있도록 한다.

### 비전 (Vision)

계획과 관리가 필요한 **모든 사람**이 자신만의 카테고리를 설정하여 계획·실행·분석의 루프를 돌릴 수 있는 범용 생산성 도구.

## 3. 핵심 기능 정의 (Scope)

### 3-1. MVP 기능

#### 대시보드

| 기능 | 설명 | 구현 상태 |
|------|------|-----------|
| 타이머 / 스톱워치 | 유저가 선택한 카테고리의 활동 시간을 측정 (카운트다운 또는 스톱워치 모드). STOP은 **일시정지**(값 유지·RESUME으로 재개)이며, 세션 기록은 RESET 또는 타이머 완료 시점에 저장(여러 번 멈췄다 재개해도 한 세션으로 합산). 측정 중·멈춤·선택 모드는 앱 재진입 시 복원. | ✅ 구현 완료 |
| 타이머 시간 설정 | 타이머 숫자 클릭 시 모달에서 시·분·초 셀을 각각 탭하여 텍스트 입력. 프리셋 칩(10·15·30분) 제공. "설정" 버튼으로 값만 적용(시작은 별도 START). 마지막 사용값 자동 기억. 스톱워치 모드 클릭 시 안내 토스트. | ✅ 구현 완료 |
| 타임테이블 | 측정한 활동 기록을 24시간 × 10분 단위 그리드에 카테고리 색상으로 표시 | ✅ 구현 완료 |
| 할 일 관리 | 카테고리별 할 일 추가·삭제·완료 토글 | ✅ 구현 완료 |
| 날짜 설정 | 날짜별로 다른 계획을 세울 수 있도록 날짜 선택 기능 제공 | ✅ 구현 완료 |

#### 분석

| 기능 | 설명 | 구현 상태 |
|------|------|-----------|
| 종합 점수 | 회고 데이터 기반 기간별(7일/30일) 평균 점수 산출, 이전 기간 대비 증감 표시 | ✅ 구현 완료 |
| 카테고리별 트렌드 그래프 | 카테고리별 일별 활동 시간(분)을 멀티라인 차트로 시각화 | ✅ 구현 완료 |
| 인사이트 | 트렌드 데이터에서 패턴(요일별 변동, 카테고리 상승/하락, 미활동 카테고리, 회고 기록률)을 자연어 카드로 제시 | ✅ 구현 완료 |
| 처방전 | 종합 점수·최약 카테고리·최약 요일을 종합하여 맞춤 개선 방안 제시 | ✅ 구현 완료 |

> **기존 학습 성향 테스트(7문항 → 페르소나 판별)는 v2.0에서 제거됨.** 유저가 직접 축적한 회고 데이터와 활동 시간을 기반으로 분석한다.
> **카테고리별 트렌드의 지표**: 활동 카테고리(과목 등)별 **일별 활동 시간(분)**. 회고 점수는 항목별로 세분된 카테고리 점수가 없으므로, 활동 카테고리 단위로는 타임테이블 history에서 집계한 활동 시간을 사용한다.

#### 설정

| 기능 | 설명 | 구현 상태 |
|------|------|-----------|
| 카테고리 관리 | 카테고리 추가·삭제, 이름 변경, 색상 설정 (대시보드 할 일·타임테이블 표시에 사용) | ✅ 구현 완료 (UI 상 "과목"으로 표시, 코드 리네이밍 예정) |
| 회고 항목 설정 | 사용자가 회고 평가 항목을 직접 추가·삭제·수정 | ❌ 미구현 |
| 하루 회고 | 목표 달성률(자동 계산) + 사용자 정의 항목(각 수동 입력) = 총점 | ✅ 부분 구현 (현재 고정 항목, 사용자 정의 전환 예정) |
| 습관 (위클리 + 매일 반복) | 요일별·매일 반복할 할 일과 플랜을 등록. 우선순위: 매일 < 요일 < 사용자 수정. **시드 트리거**: ① 그날 대시보드 첫 진입 (lazy) ② 습관 편집 직후 ③ 새 플랜 탭 생성 시 ④ 기록 탭 → 플랜 탭 전환 시. 시드 대상은 **모든 비-습관 플랜 탭**이며 시드 로그는 탭별로 분리되어 재시드를 차단한다. | ✅ 구현 완료 |

#### 캘린더

| 기능 | 설명 | 구현 상태 |
|------|------|-----------|
| 월별 캘린더 | 회고 점수와 오늘 날짜 표시, 월 단위 네비게이션 | ✅ 구현 완료 |

| 키 | 내용 |
|----|------|
| `switme_tasks` | 할 일 목록 |
| `switme_history` | 활동 세션 기록 |
| `switme_subjects` | 카테고리 목록 |
| `switme_reflection_items` | 사용자 정의 회고 항목 목록 |
| `switme_reflections` | 날짜별 하루 회고 |
| `switme_analysis` | 분석 결과 이력 |

| `switme_timer` | 타이머/스톱워치 상태 — 측정 중·멈춤(일시정지)·선택 모드 (앱 재시작 복원용) |
| `switme_timer_last_duration` | 타이머 마지막 설정 시간(초). 다음 모달 진입 시 초기값으로 사용. |

## 5. 화면 구조

PC(데스크탑)와 모바일 두 가지 셸(Shell)로 분리된 레이아웃. 동일한 state를 공유하며, 디바이스에 따라 자동 전환된다.

### 5-1. PC (Desktop Shell)

```
┌─────────┬──────────────────────────────────────┐
│ Side    │ Top Bar (제목 / 날짜 / 몰입모드)       │
│ Nav     ├────────────┬──────────┬───────────────┤
│         │ 할 일 목록  │ 타이머   │ 타임테이블     │
│ (68px)  │ + 하루 회고 │ (히어로) │ (24h 그리드)  │
│         ├────────────┴──────────┴───────────────┤
│ 탭:     │ [캘린더] [분석] [설정] — 탭 전환으로 표시 │
│ 대시보드 │                                        │
│ 캘린더   │                                        │
│ 분석     │                                        │
│ 설정     │                                        │
└─────────┴──────────────────────────────────────┘
```

### 5-2. 모바일 (Mobile Shell)

```
┌──────────────────────────────────┐
│ 헤더 (날짜 / 몰입 버튼)             │
├──────────────────────────────────┤
│ 타이머 히어로 (SVG 링 + START)     │
│ 탭 네비게이션 (4버튼) + RESET       │
├──────────────────────────────────┤
│ 콘텐츠 영역 (대시보드 기준)          │
│ ┌──────────────┬───────────────┐ │
│ │ [할일][회고]  │ 타임테이블      │ │  ← 좌·우 가로 분할
│ │  ▔▔▔         │ ─────────────  │ │
│ │  (컬럼 탭)    │ [기록][플랜1][+]│ │  ← 탭이 곧 종류 (기록/플랜)
│ │              │ ─────────────  │ │
│ │  할 일 목록   │ 24h × 10분 그리드│ │
│ │   또는       │                │ │
│ │  회고 입력    │                │ │
│ └──────────────┴───────────────┘ │
│ (캘린더/분석/설정 패널은 탭으로 전환) │
└──────────────────────────────────┘
```

> **모바일 레이아웃 규칙 (할 일 + 타임테이블이 함께 노출되는 모든 화면에 적용)**
> - 좌측 컬럼: `[할 일] [회고]` 컬럼 탭으로 전환 (`m-col-tabs`).
> - 우측 컬럼: 타임테이블 자체 탭바(기록 탭 + 플랜 탭, 각 종류 최소 1개 보장) + 24h 그리드.
> - 좌·우는 **가로 분할**을 유지한다 (모바일이라고 세로 적층/컬럼 탭 단일화로 바꾸지 않는다).
> - 신규 화면(예: 설정 > 습관)도 이 패턴을 따른다. 단, 화면 목적에 따라 좌측의 회고 탭은 숨길 수 있다.

### 5-3. 공유 컴포넌트

| 컴포넌트 | 설명 |
|----------|------|
| Zen Mode Overlay | 전체화면 몰입 모드 (카테고리명 + 대형 타이머) |
| Task Modal | 할 일 추가 모달 (카테고리 선택 + 내용 입력) |

## 8. 리스크 및 제약사항

| 항목 | 설명 |
|------|------|
| 웹 기반 한계 | 백그라운드 상시 실행, 센서·하드웨어 접근 등 일부 네이티브 기능 사용 불가. **푸시 알림은 사용 가능** (안드로이드 Chrome은 지원, iOS는 Safari 16.4+ **설치형 PWA 한정** 지원) |
| 데이터 영속성 | `localStorage` 기반이므로 브라우저 데이터 삭제 시 모든 데이터 유실 |
| 서버 미사용 | 기기 간 데이터 동기화 불가, 백업 기능 없음 |

> **정정 이력 (2026-07-15):** 기존 "푸시 알림 등 네이티브 기능 사용 불가" 기술은 사실이 아니었다. Web Push는 안드로이드에서 오래전부터, iOS에서도 16.4(2023)부터 설치형 PWA에 한해 지원된다. 이 오기재로 인해 **알림 기반 리텐션 설계가 검토 대상에서 제외되어 있었다.** 향후 리텐션 기능 기획 시 푸시는 현 스택으로 구현 가능한 선택지로 취급한다.

## 9. 수익 모델 ☐ 미착수 (BM 확정 2026-07-23 — §9-7)

> **상태:** 검토 완료. **현 구조(서버·계정 없음)에서는 유료화 자체가 불가능**하므로 로그인·백엔드 도입이 선행 조건이었다. Phase 3에서 백엔드(Firebase)·이중 인앱결제(Google Play Billing·Apple StoreKit) 도입이 **확정**되어 결제 구현이 기술적으로 가능해졌다 — 결정 근거는 §9-5.
>
> **⚠️ 목적 유의 (2026-07-21 개정):** 결제를 붙이되 목적은 **학습(앱 개발 전 과정 경험)**이다. 다만 **샌드박스에서 실결제로 전환**되었다 — 정산·수수료·세금·환불이 학습 대상에 포함되기 때문. 결정은 §9-6. 성공 기준은 매출액이 아니라 **각 과정의 완주 여부**이므로 §10-5 뒤집기조건 #1은 여전히 미발동(§10-2 판정 참조).
>
> **BM 확정 (2026-07-23):** 수익 모델을 **토큰제 Freemium**으로 확정했다. 분석 기능을 토큰으로 잠그고, 토큰은 ①보상형 광고(AdMob) ②구독(월/년, 지급형) ③토큰팩 직접 구매(consumable IAP)로 획득한다. 결정·근거·리스크는 §9-7.

### 9-1. 현 구조의 제약

| 제약 | 수익화에 미치는 영향 |
|---|---|
| 서버 없음, `localStorage` 단독 | 결제 검증·구독 상태 확인 불가 |
| 기기 간 동기화 불가 | 유저가 지불할 가장 명확한 이유(데이터 안전·기기 이동)가 부재 |
| 도메인별 `localStorage` 분리 | 배포 주소가 다르면 데이터도 분리 (§개선사항목록 기록 참조) |

### 9-2. Freemium 구조 — 무료/유료 경계 (§9-7에서 토큰제로 확정)

> **상태:** 아래는 후보 검토 단계의 경계 구분이다. **2026-07-23 §9-7에서 토큰제 Freemium으로 확정**됐다 — 분석 전체가 토큰 뒤로 이동. 확정 후 경계는 §9-7을 따른다.

무료 기능은 유지하고, **데이터가 축적될수록 가치가 커지는 것**만 유료로 잠근다.

| 구분 | 범위 |
|---|---|
| **무료 (확정)** | 타이머·타임테이블·할 일·습관·캘린더·회고 = **기록·실행 기능 전부** |
| **유료 — 토큰 소비 (확정)** | 분석 **전체** (7일·30일·90일·연간 트렌드·기간 비교) + 데이터 내보내기(CSV·이미지 리포트). 깊이별 토큰 차등 → §9-7 |
| **동기화·백업** | 무료 제공(데이터 안전은 기본 가치). 유료 라인 아님 |
| **처방전 심화 (후순위)** | 규칙 기반이 아닌 개인 이력 축적 기반. 추후 토큰 소비처로 편입 검토 |

**근거:** 타이머·할 일은 대체재가 많아 유료화 시 즉시 이탈한다. 반면 분석은 스윗미의 차별점(회고→분석→처방 루프)이자 유저 축적 데이터 기반이라 **대체재가 적어** 유료화해도 이탈 압력이 낮다. 그래서 7일 분석까지 토큰 뒤로 옮긴다(§9-7).

### 9-6. 실결제 전환 결정 (2026-07-21)

### 9-7. BM 확정 — 토큰제 Freemium (2026-07-23)

**결정:** 수익 모델을 **토큰제 Freemium**으로 확정한다.

- 기록·실행 기능(타이머·타임테이블·할일·습관·캘린더·회고)은 **전부 무료**.
- **분석 기능 전체를 토큰으로 잠근다** (7일 분석 포함 — 기존 무료에서 이동).
- **매달 무료 토큰 1개 기본 지급** (전 유저 — 결제·광고 없이도 월 1회 7일 분석 가능. 진입장벽 0 유지).
- 추가 토큰 획득 **3라인**: ① **보상형 광고**(AdMob) ② **구독**(월/년, 지급형 — 기간마다 토큰 정액 지급) ③ **토큰팩 직접 구매**(consumable IAP).

**토큰 이코노미 (초안 — 수치 조정 가능):**

| 소비처 | 토큰 |
|---|---|
| 7일 분석 | 1 |
| 30일 분석 | 3 |
| 90일/연간 트렌드 | 5 |
| 리포트 내보내기(CSV·이미지) | 2 |

획득: **무료 기본 = 매달 1토큰 자동 지급(전 유저)** · 보상형 광고 1회 = 1토큰(일 상한으로 남용 차단) · 구독 월형 = 월 정액 대량 지급 · 연형 = 월형 대비 할인(통상 2개월분) · 토큰팩 = 개수별 consumable 상품.

> **무료 월 1토큰 근거:** 무료 유저도 결제·광고 없이 **월 1회 7일 분석**을 체험 → 분석의 가치를 먼저 맛보고 추가 사용 시 광고·구독으로 전환. 진입장벽 0을 유지하면서 유료 전환 유인을 만든다. (수치는 초안 — 지급 주기·개수 조정 가능.)

**⚠️ 전제:** 목표는 **학습(전 과정 완주)**이며 성공 기준은 매출액이 아니라 완주 여부(§9-6·§10-2). 이 BM은 **결제 유형을 최대한 넓게 겪기 위해** 설계됐다 — 구독(subscription IAP)·토큰팩(consumable IAP)·광고(AdMob)가 전부 다른 정산 시스템. 매출 극대화 설계가 아니다.

## 10. 플랫폼 전략 — 하이브리드 앱(Capacitor) 결정 (2026-07-23)

> **⚠️ 폐기 이력 — 조용히 고치지 않는다 (원칙 §4-3·5부).**
> 본 §10은 **2026-07-15 "TWA 결정"을 폐기하고 대체**한다. 폐기 사유는 아래 §10-1이며, 구 TWA 결정의 근거·리스크는 삭제하지 않고 §10-4(기각안)에 **강등 보존**한다. 결론만 승계하지 말고, 폐기된 결정을 되살리려면 §10-1의 발동 조건을 먼저 확인하라.

### 10-1. 결정

**하이브리드 앱(Capacitor)으로 Google Play + Apple App Store 동시 진입. iOS·Android 둘 다 커버한다.**

Capacitor는 기존 PWA(HTML/CSS/Vanilla JS)를 **거의 그대로** 네이티브 셸(WebView)에 담아 iOS·Android 앱으로 빌드하는 방식이다. 코드 재작성이 없고, 한 웹 코드베이스로 양 플랫폼을 낸다.

> **폐기된 TWA 결정과의 결정적 차이:**
> - TWA는 **Android 전용**(Apple 미허용) → iOS를 못 냄.
> - TWA는 **Chrome 저장소 공유** → 유저가 브라우저 기록을 지우면 데이터 소실(구 §10-6). Capacitor는 **앱 자체 네이티브 저장소**를 써서 이 리스크가 **구조적으로 소멸**한다.

### 10-2. 결정의 전제 — ⚠️ 이것이 바뀌면 결정도 무효

> **현재 목표는 "학습"이다 (2026-07-21 정의, 2026-07-23 iOS 범위 추가):**
>
> **앱 개발의 전 과정을 직접 겪는다.** 기획·구현·출시뿐 아니라 **유지보수·홍보·매출·정산·세금·환불 대응까지 포함**한다. **성공 기준은 유저 수나 매출액이 아니라 각 과정을 완주했는가**이다.
>
> **2026-07-23 범위 추가:** 커버 대상 플랫폼에 **iOS를 포함**한다. 이로써 "스토어 운영"이 Google Play 한 곳이 아니라 **Play + App Store 이중 스토어**로 확장됐다 — 두 스토어의 심사·결제·정산 차이를 겪는 것이 학습 산출물에 포함된다.

> **학습 범위 확장 이력**
>
> - **2026-07-19 (1차):** 로그인·계정 동기화·인앱결제(IAP) 연동 흐름을 학습 대상에 포함(Phase 3). 이 시점엔 **결제를 샌드박스로 한정**했다.
> - **2026-07-21 (2차):** 샌드박스는 **돈·법·세금이 빠진 버전**이라 "전 과정 경험"에 미달함이 드러났다. **실결제·공개 출시·홍보**를 범위에 포함한다. 결정은 §9-6.
> - **2026-07-23 (3차):** **iOS 지원**을 요구로 추가. 구 TWA 결정의 뒤집기조건 #2(iOS 필요)가 발동 → 플랫폼 전략을 하이브리드(Capacitor)로 교체.

> **뒤집기조건 #1(학습→제품 전환) 재판정 (2026-07-21, 2026-07-23 유효): 미발동.**
>
> | # | 근거 |
> |---|---|
> | 1 | **수익이 목적이 아니라 학습 소재다.** 실결제를 켜는 이유는 정산·수수료·세금·환불이 어떻게 도는지 겪기 위함이며, 매출은 부산물이다 |
> | 2 | **성공 기준이 숫자가 아니다.** 홍보를 하더라도 판정 기준은 "홍보→유입→매출→환불→세금 신고를 한 바퀴 겪었는가"이지 "몇 명 모았는가"가 아니다. 제품 목표라면 숫자가 기준이 된다 |
> | 3 | **트리거가 격상시킬 리스크가 이미 해소 예정이다.** #1의 실효는 "데이터 유실이 감내 불가로 격상되고 서버가 필수가 된다"인데, Phase 3에서 서버 도입이 이미 확정이다 |
>
> **철회된 근거:** 초안에서 근거 ②로 "유저 확보 동기 없음 — 홍보 행동이 추가되지 않았다"를 썼으나, 홍보가 학습 범위에 포함되면서 사실과 어긋나 **명시적으로 철회**했다(원칙 §4-3). 위 표 2번이 대체 근거다.

| 대안 | 사유 |
|---|---|

### 10-5. 결정을 뒤집어야 하는 조건 (Reversal Triggers)

아래 중 하나라도 발생하면 **§10 전체를 재검토**한다. 결론만 승계하지 말 것.

1. **목표가 "학습" → "제품(유저 확보·수익)"으로 변경.** 확장성·비용·성능 기준이 새로 지배하며, WebView 성능 한계가 병목이 되면 풀 네이티브 재검토가 열린다.
2. **WebView로 감당 못 하는 네이티브 성능/기능이 핵심 요구가 됨.** (고빈도 애니메이션, 상시 백그라운드 처리, 특수 센서 등) → 풀 네이티브 전환 검토.
3. **한쪽 스토어 정책이 하이브리드 WebView 앱을 배제.** (예: Apple의 "웹뷰 단순 래핑" 리젝) → 해당 플랫폼만 네이티브 또는 철수 재검토.

### 10-7. 스토어 출시 관문 — Android 의무 테스트 + iOS 심사

> **Android:** 유저 계정은 **개인 계정**이라 구글 정책상 비공개 테스트가 **선택이 아니라 강제 경로**다.

- **Google Play (개인 계정, 2023-11-13 이후 생성):** 공개 출시 전 **비공개 테스트에서 테스터 12명 이상 × 14일 연속**을 충족하고 프로덕션 권한을 신청(심사 최대 7일)해야 공개가 열린다. 회피 불가·의무.
  - 결과적으로 별점 리스크는 자동 회피되나 그건 선택의 결과가 아니다.
  - 목표 정의(§10-2)에 홍보·매출·유저 반응·별점 관리가 포함되므로 비공개 종료는 목표 미달 → **공개 출시까지 확정**. 근거 §9-6.
- **Apple App Store:** 12명×14일 같은 테스터 관문은 **없다.** 대신 **모든 빌드가 심사**를 거치며(통상 1~3일, 유료/기능 앱은 더 엄격), **TestFlight**로 베타 배포한다. 즉 Android보다 관문은 가볍지만 심사 재량·거절 리스크는 더 크다(§10-6).
- **결제 학습에 관한 사실:** 인앱결제 **기술 흐름**(상품 등록·결제창·구매 검증·기능 해제)은 양 스토어 모두 **샌드박스/테스터로 완주 가능**하다. "결제를 배우려면 공개가 필요하다"는 연결은 성립하지 않는다. 공개 출시를 요구하는 것은 결제가 아니라 **실제 정산·세금·낯선 유저 반응·공개 별점**이다.
- **정책 출처:** Play Console Help — App testing requirements for new personal developer accounts (2024-12-11: 테스터 20→12명). Apple — App Store Review Guidelines, TestFlight.
