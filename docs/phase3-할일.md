### ☐ 그 다음 — §2 유저 실검증 (완료 조건, Claude 대행 불가)
1. **충돌 루프 해소 확인** — 로그인 시 충돌 창이 **한 번만** 뜨고 반복 안 되는지 (세션당 1회 수정 검증)
2. **게스트 누수 해소** — 계정 A 로그인→할일 설정→로그아웃 시 게스트에 그 할일이 **안 남는지**
3. **재로그인 복구** — 계정 A 할일→게스트 전환·삭제→재로그인 시 할일이 **복구되는지**
4. **2-6 기기 2대 교차** — PC서 할일 생성 → 폰서 같은 계정 로그인 시 뜨는지, 역방향도
5. **2-7 유실 복구** — 브라우저 사이트 데이터 삭제 후 재로그인 시 복구되는지 → PRD §10-6 유실 리스크 소멸 검증
6. 통과 시 §2 완료 처리(2-6/2-7 ✅) → **§3 푸시 알림** 착수

### ⚠️ 알려진 잔여 UX (버그 아님, 필요 시 다듬기)
- 같은 기기라도 **새 세션(새 탭/다음날)** 첫 로그인 시 충돌 창이 1회 뜰 수 있음 — 앱 로딩 시 습관 시드 등 정규화가 로컬을 미세 변형시켜 클라우드와 "다름"으로 잡히기 때문. 무한 반복은 수정됨(세션당 1회 제한). 거슬리면 정규화-차이 무시 로직 추가 검토.
- **충돌 선택 = 되돌리기 없는 전체 덮어쓰기** (미수정, 2026-07-24 점검서 식별). 충돌 창에서 한쪽 선택 시 반대쪽 데이터가 통째로 소멸하며 병합·백업·되돌리기가 없다. B+ 설계상 의도된 동작이나, 오클릭 시 유실이 크므로 덮어쓰기 직전 반대쪽 임시 스냅샷(되돌리기 여지) 추가를 향후 검토.

## ⚠️ 순환 의존 — Phase 3는 출시를 사이에 두고 쪼개진다

"Phase 3 전부 완료 → 출시"는 **물리적으로 불가능**하다. §4(결제)의 인앱 상품 등록은 **Play Console에 앱이 생성돼 있어야** 가능하므로, **출시가 결제의 선행 조건**이다.

```
§P 사전준비 → §1 로그인 → §2 동기화 → §3 푸시     ← [출시 전]
        ↓
[출시 — docs/하이브리드-출시-할일.md A·B단계 (Android 비공개 + iOS TestFlight)]  ← 양 스토어에 앱 생성
        ↓
§4 결제 구현·검증                                  ← [출시 후] 여기서만 가능
        ↓
[테스터 12명×14일 → 공개 출시 + 실결제 개시]
        ↓
§5 배포 시 갱신 의무
```

**의도 보존:** 공개 출시 시점엔 §2가 끝나 **데이터 유실 문제가 없다.** 이것이 "Phase 3 먼저" 결정의 목적이었다(`docs/로드맵.md`).

- [ ] ☐ **4-2** 분석 기능 **토큰 게이팅** — 7일=1 / 30일=3 / 90일·연간=5 / 리포트=2 (초안, PRD §9-7). 잔량 부족 시 획득 안내(광고·구독·토큰팩)
- [ ] ☐ **4-3** 토큰 획득/차감 트랜잭션 로직 + 잔량 UI(무료/유료 셸 양쪽)
- [ ] ☐ **4-3b** **매달 무료 토큰 1개 자동 지급** — 전 유저 대상, 월 1회 지급 스케줄(중복 지급 방지 시점 관리). 근거 PRD §9-7

### 4-B. 인앱결제 (IAP — 구독 + 토큰팩)

- [ ] ☐ **4-4** 가격 책정 — 각 상품 **최저가**. 구독(월/년)·토큰팩(개수별) 양 스토어 가격 티어 각각 설정
- [ ] ☐ **4-5a** Play Console에 인앱 상품 등록 (Android) — 구독 상품 + consumable 토큰팩
- [ ] ☐ **4-5b** App Store Connect에 인앱 상품 등록 (iOS) — 구독 상품 + consumable 토큰팩
- [ ] ☐ **4-6** Capacitor 인앱결제 플러그인 연동 — Android는 Google Play Billing, iOS는 Apple StoreKit 호출. (구 TWA Digital Goods API 방식 폐기 — PRD §9-4)
- [ ] ☐ **4-7** Firebase로 구매 검증(양 스토어 영수증) → 구독=정기 토큰 지급, 토큰팩=즉시 토큰 지급
- [ ] ☐ **4-8** **환불 처리 흐름 구현** — 구독 취소·토큰팩 환불 시 토큰 회수·기능 재잠금

### 4-C. 보상형 광고 (AdMob)

- [ ] ☐ **4-9** AdMob 계정 생성 + Capacitor AdMob 플러그인 연동 (양 플랫폼)
- [ ] ☐ **4-10** 보상형 광고 시청 완료 → 토큰 1개 지급. **일 획득 상한** 설정(남용 차단)
- [ ] ☐ **4-11** 광고는 **자발 시청만** — 몰입 화면 강제 노출 금지(PRD §9-7 정체성 회피 조건)

### 4-D. 유저 수동 검증 (완주 = 목표)

- [ ] ☐ **4-12** (유저) **샌드박스 검증 먼저** — Google 라이선스 테스터 / Apple Sandbox 테스터로 실결제 전 기술 흐름 확인 (테스트 수단으로는 계속 사용)
- [ ] ☐ **4-13** (유저) **실결제 1회 완주** — 구독·토큰팩 각각, 스토어별. 결제 → 입금 → 수수료 차감 확인
- [ ] ☐ **4-14** (유저) **환불 1회 완주** (스토어별) — 요청 접수 → 처리 → 토큰 회수·기능 재잠금 확인
- [ ] ☐ **4-15** (유저) 정산 내역·세금 신고 자료 확인 — IAP(Google·Apple) + **AdMob 광고 수익(별도 정산)** 각각

> **샌드박스는 폐기가 아니라 강등이다.** 목표 달성 수단에서 **개발 중 테스트 수단**으로 역할이 바뀌었다(4-12).
