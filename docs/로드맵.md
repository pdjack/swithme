# 스윗미(swithme) — 개발 로드맵

> **문서 성격**: 단계별 마일스톤·기술 구조 요약. 결정 근거·기획 상세는 `docs/PRD.md`(§9 수익모델·§10 플랫폼 전략)를 단일 진실원으로 참조한다.
> **기간 산정 전제**: AI 활용(Claude Code) 코딩 가속 + 일 2시간 가용. 순수 개발 구간은 압축되나, **외부 관문(구글 비공개 테스트 14일 등)은 AI로 단축 불가**하므로 별도 표기한다.

## 2. 개발 단계별 마일스톤

> SwitMe MVP는 이미 완성 상태 → Phase 1~2는 완료. 기간은 **AI 활용 기준으로 재산정**했다(원본 템플릿의 1~12주 → 압축).

| 단계 | 기간(AI 기준) | 마일스톤 | 핵심 목표 (Deliverables) | 상태 |
|---|---|---|---|---|
| Phase 1 | ~1주 | Core Logic | 데이터 구조(localStorage `state`), 타이머·타임테이블·회고 집계 알고리즘 | ✅ 완료 |
| Phase 2 | ~1~2주 | MVP UI/UX | 대시보드·분석·설정·캘린더 화면, PC/모바일 셸, 화면 전환 | ✅ 완료 |
| Phase 3 | ~2~3주 | Features | 로그인·계정 동기화·백업, 푸시 알림, 인앱결제(IAP). **스택 확정: Firebase(iOS·Android SDK 지원) + 이중 IAP(Google Play Billing·Apple StoreKit)** (PRD §9-5). **결제는 실결제·상품 1개·최저가** (PRD §9-6). 세부 → `docs/phase3-할일.md` | ☐ 미착수 (스택 확정) |
| Phase 4 | ~2~3주 개발 **+ 14일 관문(Android, 고정)** | Beta/Release | **하이브리드 앱(Capacitor)으로 Google Play + Apple App Store 동시 진입** — Android는 비공개 테스트(개인 계정 의무) 12명×14일 → 프로덕션 심사, iOS는 TestFlight → App Store 심사 → **공개 출시 확정**. 실결제에 따른 판매자 등록·사업자 확인·환불 방침 포함. 세부 → `docs/하이브리드-출시-할일.md` | ☐ 진행 예정 |
| Phase 5 | 상시 | 운영·홍보 | 자체 제작 콘텐츠 홍보(`docs/마케팅플랜.md`), 별점·리뷰 대응, 환불 응대, **양 스토어 매년 타겟 SDK 재빌드**, 세금 신고 | ☐ 신규 (2026-07-21) |

### AI가 줄이는 구간 / 못 줄이는 구간

- **AI로 단축:** 코드 구현, 테스트 작성, 리팩토링, 문서 동기화 → Phase 3 개발 구간이 주 단위로 압축.
- **AI로 단축 불가(고정 관문):**
  - 구글 개인 계정 비공개 테스트 **12명 × 14일** — 정책상 강제, 코딩 속도 무관.
  - 실기기 QA·유저 피드백 수집 기간.

### 순서 제약

- **Phase 3(서버·계정)이 수익화(§9)의 선행 조건.** (데이터 유실은 하이브리드 네이티브 저장소로 이미 해소 — PRD §10-6.)
- 하이브리드 출시(Phase 4)는 Phase 3 없이 착수 가능. **데이터 유실 리스크는 하이브리드 셸 자체가 제거**하므로 서버 이전에도 잔존하지 않는다(PRD §10-6, 2026-07-23 갱신).
- **실결제(Phase 3 §4)는 Phase 4 D단계(사업자 확인·판매자 등록·환불 방침)가 선행 조건.** 기술이 아니라 행정이 병목이므로 **D-1(사업자·통신판매업 확인)은 지금 병렬 착수가 유리**하다.

### ✅ 출시 순서 (2026-07-21 확정, 2026-07-23 근거 일부 변경) — Phase 3 먼저

**결정:** Phase 3(로그인·동기화·푸시)를 먼저 하고 출시한다. 개인정보처리방침은 처음부터 "수집함" 기준 1회 작성.

> **⚠️ 전제 변경 (2026-07-23):** 이 순서의 **1순위 근거였던 "데이터 유실 해소"는 이제 하이브리드 셸이 독립적으로 제공**한다(네이티브 저장소, PRD §10-6). 따라서 Phase-3-먼저의 근거는 **"방침을 '수집함'으로 1회만 작성(2회 재작성·재제출 회피)"와 정직성**으로 이동했다. 데이터 유실 방지는 더 이상 순서를 강제하지 않는다. → **순서를 유지할지 재검토할지는 유저 판단 대상**(발견만 기록, 미실행).

> **⚠️ 순환 의존(불변):** "Phase 3 전부 완료 → 출시"는 **물리적으로 불가능**하다. Phase 3 §4(결제)의 인앱 상품 등록은 **스토어에 앱이 생성돼 있어야** 가능하므로, 출시가 결제의 선행 조건이다. 따라서 Phase 3를 **출시 전/후로 쪼갠다.**

```
Phase 3 ① 로그인 → ② 동기화·백업 → ③ 푸시            ← 출시 전
        ↓
[Android 비공개 테스트 출시 + iOS TestFlight]         ← 여기서 양 스토어에 앱 생성
        ↓
Phase 3 ④ 결제 구현·검증 (Google IAP + Apple IAP)     ← 상품 등록은 여기서만 가능
        ↓
[Android 테스터 12명×14일 / iOS 심사] → [양 스토어 공개 출시 + 실결제 개시]
```

**기각한 순서 (출시 먼저):** 출시가 2~3주 빠르나 방침·데이터 안전 양식을 2회 작성해야 한다. "방침 갱신·재제출 경험"은 출시 후 기능 추가 시 자연히 발생하므로 **영구 손실이 아니라 시점 이동**이다.

## 3. 기술 스택 및 구조

### 프론트엔드 / 백엔드

- **프론트엔드:** HTML5 + Vanilla JS(ES6+) + Vanilla CSS. PWA(`vite-plugin-pwa`, Service Worker).
- **앱 셸:** ✅ **확정 — 하이브리드(Capacitor)**. 기존 웹 코드를 네이티브 WebView에 담아 iOS·Android 동시 빌드. 결정 근거·기각안·뒤집힘 조건은 PRD §10.
- **백엔드:** ✅ **확정 — Firebase(BaaS)**. 인증(Firebase Auth: 구글+이메일), DB·동기화(Firestore), 푸시(FCM), 결제 검증을 통합 처리. Firebase는 iOS·Android SDK를 정식 지원. 결제는 **이중 IAP**(Google Play Billing + Apple StoreKit). 결정 근거·기각안·뒤집힘 조건은 PRD §9-5.

### 모듈화 계획 (UI ↔ 비즈니스 로직 분리)

- **상태 레이어:** `js/store.js`의 `state` 객체 중앙 관리, 변경 후 `saveToLocal()` 호출 필수.
- **로직 레이어:** 타이머 로직 `js/timer.js` 독립. 새 전역 함수는 `window` 등록 패턴.
- **UI 레이어:** PC(`#desktop-shell`)/모바일(`#mobile-shell`) 셸 분리. 모바일은 PC 타이머를 `MutationObserver`로 감시·동기화.

### 데이터 모델

- **현재(클라이언트 단독, localStorage):** 카테고리, 날짜별 할 일/플랜, 타임테이블 history(활동 세션), 회고 점수, 습관(요일·매일 반복).
- **사용자/계정 데이터:** ☐ Firestore 스키마 **설계 예정**(Phase 3) — 스택은 Firestore로 확정, `state` 구조 매핑 설계는 3.2에서. 세부 → `docs/phase3-할일.md`.
